Security runtime events

 

This topic describes the security runtime events and performance counters generated by .

administrators need to monitor the health of the system and take action on system failures. System health monitoring tools rely on failure events and counters generated at runtime by the platform to assess the state of the system.

This topic describes the security runtime events and performance counters generated by .

Access denied performance counters

will keep a counters of the number of access denied events per minute to allow monitoring and identifying denial of service attacks.

The following counters are provided:.

  • "Operation access denied" events per minute (global for all services)
  • "Data access denied events" per minute (global for all services)

Unlogged security events

The events in this section are not logged and are generally the client application's responsibility.

Access denied events

Amalga does not log access denied events because they are as it is a routine security operation. If client applications are denied access to resources for which they are authorized, the client application is responsible for logging that event.

Failed queries

does not log events for failed queries. The data connection takes the query as passed by an application. It is the responsibility of the client application to log a failed query event.

Missing relying party trust

Amalga does not log events for missing relying party trusts. If a client application is supposed to be a relying party but misses relying party trust, it is the responsibility of the client application to log such events.

Resource does not exist

Amalga does not log failure events for missing resources. It is the client application's responsibility to log such events.
