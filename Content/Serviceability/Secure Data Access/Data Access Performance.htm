Data access performance

 

This topic describes how to optimize the performance of data access.

Data access performance is gated on performance. Follow optimization best practices for SQL statements when writing rule filters (SQL conditions) and binding entities to data.

Materialized entity views are automatically generated when changes to entity binding are saved. For more information about materialized entity views, see Materialized entity view.

This section lists specific best practices by area.

Area Best practice
Indexing Ensure that all physical table columns used in access rules are indexed.
Binding SQL statement
  • Limit the number of joins in the view. For improved performance, do not use joins, if possible.
  • Use LEFT JOIN statements if a JOIN statement must be used in SQL views and entity bindings.
  • Avoid using UNION statements in SQL views and entity bindings.
Roles Minimize the overlap among roles for best performance and manageability. That is, if a user is a member many roles, the time required to access data is expected to be increased, compared with a user who is a member of only one role.
Entity properties Minimize the number of properties defined in an entity to only those required by the rules contained in the access policy. You can always add properties on an as-needed basis.
Sensitive labels Minimize combined sensitive labels. A single sensitive label can be applied to multiple entity properties within a schema. Minimizing labels required for manageability of the solution improves the performance of .
Rule filters Minimize the number of entity properties used in rule filters. For instructions on debugging access rules, see Debug access rules.
