Logical architecture

 

This topic describes the conceptual software architecture which supports application development.

The logical view of architecture contains the main service layers and their key functions. Not all of the layers are required to create a useful system. Taken together, the application services and platform services layers form the foundation for application runtime execution. With those layers in place, the services exist to support an hosted application.

If an enterprise wants to standardize on a delivery mechanism for its applications, the layer can be added to provide specialized support in addition to the standard capabilities.

Platform services

The platform services layer supports all the system, data access, and application management functions. This layer forms the foundation of , and its services are primarily exposed by endpoints. For more information on these endpoints, see Service Endpoint Reference.

introduces the concept of an topology. A topology is the set of services, applications, features, logical hosts and physical hosts that are registered in the database (the main topology store). One of the primary functions of platform services is to manage the installation and configuration of all the topology components.

Other major functions of the platform services layer include the following:

  • Logging of operational, performance, and auditing data
  • Support for the entity registry and virtual schema abstraction layers
  • Authentication and authorization
  • Data access support for ADO.NET, OLE DB, and AmalgaDataSource
  • Topology web service record of everything that is deployed and managed as part of the topology
  • Registration web service is used to register new features or applications
  • Discovery web service provides service configuration and topology registration information
  • Central configuration service provides a method to store configuration information for any requesting service

Application services

The application services layer provides the runtime environment for an hosted application. For more information on the definition of an hosted application, see Hosted applications. Because application services can be installed on servers independently from platform services, some capabilities such as data access support are in both platform and application services.

The application services layer contains an application hosting platform that enables the server to deploy and host services supporting hosted applications. It includes special .NET libraries, applications and functionality.

By including in the topology, an enterprise can take advantage of standard capabilities such as site collections, list management, and document libraries. In addition, provides additional support for applications authored in .

  • A collection of -specific controls which enable development. The controls include validators, patient-oriented functions, gridviews, and data sources.
  • Runtime support for extended functions such as context sharing between controls.
  • A basic portal framework including the following sites:
    • Patient: Used by the enterprise to communicate to patients, family, and caregivers.
    • Internal Provider: Used by providers inside the institution to view information about their patients.
    • Staff: Used by the institution staff (billing or registration staff) to respond to patient and provider inquiries.
    • Public: Used to provide an experience for an unauthenticated user.

Application layers

The two application layers in the logical architecture diagram correspond to applications that are either running directly on top of the platform (platform and application services layer), or on top of . The applications can include those provided by , and also third party or customer developed applications.

For more information on application development, see Developing an Amalga application.

Management services

The platform services layer includes the infrastructure to manage the topology. Layered on top of that infrastructure is an administrative tool called the . This graphical interface supports administrators in managing the properties of the following components.

  • System Components
    • Features
    • Logical hosts
    • Physical hosts
    • Data hosts
  • Hosted applications
    • Services supporting hosted applications
  • Data entities
    • Bindings
    • Access policies
    • Sensitive labels
