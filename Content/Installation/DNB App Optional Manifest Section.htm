Eventing installation configuration

The eventing service manifest's settings are embedded as a character data string. Make the desired changes to the elements within that string.

Eventing has four service files.

File Name Action
EventingChangeTrackerService Optional configuration
EventingDispatcherService Optional configuration
EventingManagerService Optional configuration
EventingWcfHandlerService No changes accepted

EventingChangeTrackerService manifest

Navigate to the <DefaultValue> node and make the desired changes to the services and properties contained in the character data string and defined in the following tables.

<DeploymentManifest>
     <DoorsWorkerManifest>
     <Settings>
          <Setting>
               <DefaultValue>

Repository service

Property name Description
CommandTimeout SQL Server Timeout in seconds
RetryCount Number of attempts to read/write data from SQL Server in case of deadlocks/timeouts
RetryInterval Time interval in milliseconds to wait before the next retry

Change-tracker service

Property name Description
PollingInterval How often change tracker service polling database server for data changes (in seconds)
CacheTimeoutInterval Change tracker service keeps a copy of resolved SQL statement to compute data changes. This statement reflects the current user’s entity binding and authorization on entity. This parameter keeps track of how long we will keep the resolved SQL statement before refreshing.
MaxAvailableEvents Maximum number of events that can be process by a single eventing application at one time
MetadataReaderPollingInterval How often DCT check for update of Event definition in metadata DB, polling interval in seconds
CommandTimeout SQL Server CDC query and change result query Execution Timeout in seconds
MaxChangeProcessorThreads Sets the number of requests to the change tracker thread pool that can be active concurrently. All requests above that number remain queued until thread pool threads become available. This setting is multiplied by number of processors to get the actual number of threads
MaxConcurrentQueries Throttling mechanism to limit the number of calls (absolute number) sent to SQL Server
MaxExecutionTime Max allowed execution time that an event processing should take in seconds. The events will get suspended if it exceeds this time for more than MaxTimeoutRetries times
MaxTimeoutRetries If the MaxExcectuionTime is exceeded more than this many times by any event, the event will get suspended
StaleEventBuffer When a DCT service processing an event crash. The event will be marked being processed and no other DCT service will pick up. The event become stale. This parameter decide how long we will wait before DCT will reset a stale event to be active such another DCT service can pick up and process an event
MaxChangedRecords This is appended as TOP N in the query generated to get changed records. If 0 or negative number is specified, this property will be ignored
MaxCdcRecords Maximum number of CDC changes that will be fetched per table per CDC polling
BatchSize The number of notifications that are sent to the data connector for writing event notification logs into data change notification private database
BatchTimeout If the event log batch size does not hit the BatchSize in this amount of milliseconds, the logs will be sent automatically without waiting to reach the required batch size
MaxEventMesageSize Max size of the event data in bytes that can be sent to the event dispatcher in one message. If the size of the message exceeds this limit, then only the LogId is sent to the dispatcher, otherwise the data is also sent

Event-dispatcher service

Property name Description
MetadataReaderPollingInterval How often event dispatcher talk to topology service to refresh configuration
MaxDispatcherThreads Sets the number of requests to the dispatcher thread pool that can be active concurrently. All requests above that number remain queued until thread pool threads become available. This setting is multiplied by number of processors to get the actual number of threads
MaxConcurrentActions Throttling mechanism to limit the number of action invocations (absolute number)
ContinueOnError If ContinueOnError is true, then if dispatcher fails to call WCF handler for event notification, server will log the information and continue. If this is set to false, then the whole batch of events will be rolled back. Rolling back might lead to duplicates events being sent to the destination
