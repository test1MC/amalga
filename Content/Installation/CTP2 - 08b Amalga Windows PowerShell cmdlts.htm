Amalga reference

 

This is not an exhaustive reference to commands. Only selected cmdlets are included.

This topic lists cmdlets used to install and uninstall hosted applications, in addition to cmdlets used for security (such as management of roles and application permissions). For more information about procedures related to installation cmdlets, see Install applications.

Publish-ApplicationInstance

The Publish-ApplicationInstance cmdlet uploads the application package to the database.

Syntax

Publish-ApplicationInstance [-ApplicationCabFile] <string>

[-ApplicationAuxiliaryFiles] <string[]> [<CommonParameters>]

Parameter Descriptions

[-ApplicationCabFile] <string>

Takes the local or Universal Naming Convention (UNC) path for the application .cab file to be uploaded to database.

Required? Yes
Default value None
Accepted values Local or UNC path. Examples: \\sharename\TestApp.cab,

c:\Temp\TestApp.cab, .\TestApp.cab
Accept Wildcards? No

[-ApplicationAuxiliaryFiles] <string[]>

Defines the semicolon-separated UNC paths to all the resolved application and service manifest files.

Required? Yes
Default value None
Accepted values

Semicolon-separated UNC paths to the resolved application and service manifests.

  • The manifest file names should be exactly the same as the application and service names.
  • There should be one resolved manifest file for the application and one resolved manifest file for each service that is part of the application.

  • Manifest file names are case-insensitive.

  • Examples: C:\Manifest\LabApp.xml, .\LabApp.xml,

    \\sharename\LabApp.xml
Accept Wildcards? No

<CommonParameters>

This cmdlet supports the common parameters: -Debug, -ErrorAction, -ErrorVariable, -OutBuffer, -OutVariable, -Verbose, -WarningAction, and -WarningVariable. For more information, see Common Parameters.

When -Verbose is set to True, Publish-ApplicationInstance logs all displayed

information.

Example

Publish-ApplicationInstance -ApplicationCabFile "C:\Manifest\ApplicationWith1WCFService.cab" -ApplicationAuxiliaryFiles "C:\Manifest\TestApplication1.xml; C:\Manifest\TestWCFAmalgaService1.xml; C:\Manifest\TestWCFAmalgaService2.xml"

 

New-AmVirtualSchema

The New-AmVirtualSchema cmdlet takes the Entity Data Model (EDMX) file from the application .cab and passes it to the virtual schema service, which creates the for the application instance.

If the EDMX file contains read/write application entities, the New-Virtual Schema cmdlet creates the public entities, private database, and their private registry in the database.

Syntax

New-AmVirtualSchema [-ApplicationInstanceName] <string>

[-DataHostInstanceName] <string> [-force] [<CommonParameters>]

Parameter Descriptions

[-ApplicationInstanceName] <string>

Takes the unique application instance name as recorded in the topology database and included in the resolved application or service manifest used during Publish-ApplicationInstance process.

Required? Yes
Default value None
Accepted values Application instance name as recorded in the database
Accept Wildcards? No

[-DataHostInstanceName] <string>

Takes the alias name of the database server instance as recorded in the database.

For , the format of the DataHostInstanceName is "FQDNServer_Instance".

Required? Yes
Default value None
Accepted values Database server alias
Accept Wildcards? No

[-force]

This parameter takes no values. When this parameter is specified, New-AmVirtualSchema cmdlet overwrites the previously existing virtual schema for the application. A common usage of this parameter is to overwrite a corrupted that can not be removed.

Required? No
Default value None
Accepted values None
Accept Wildcards? No

<CommonParameters>

This cmdlet supports the common parameters: -Debug, -ErrorAction, -ErrorVariable, -OutBuffer, -OutVariable, -Verbose, -WarningAction, and -WarningVariable. For more information, see Common Parameters.

When -Verbose is set to True, New-AmVirtualSchema logs all information displayed on the console.

Example

New-AmVirtualSchema -ApplicationInstanceName "ApplicationNameInstance1" -DataHostInstanceName "localhost"

Sync-PhysicalHost

The Sync-PhysicalHost cmdlet queries the topology service, downloads all the new changes to the physical host and applies them. These changes include the following actions

  • Downloading new services and hosting them
  • Uninstalling services from the physical host

For Doors Worker services, if Sync-Physical Host has new changes for the service, the service needs to be in a Stopped state before running Sync-PhysicalHost.

Syntax

Sync-PhysicalHost –ServiceInstanceName <string> -force [<CommonParameters>]

Parameter Descriptions

[-ServiceInstanceName] <string>

Takes the unique instance name as recorded in the topology database and included in the resolved application or service manifest used during the Publish-ApplicationInstance process.

Required? No
Default value None
Accepted values Application instance name as recorded in the AmalgaSystem database
Accept Wildcards? No

[-force]

This parameter takes no values. When this parameter is specified, then services are forced to synchronize on the physical host, regardless of the requirement.

Required? No
Default value None
Accepted values None
Accept Wildcards? No

Examples

Sync-PhysicalHost

Sync-PhysicalHost -ServiceInstanceName TestApp1 –force -Verbose

Get-AmConfigurationSetting

Get-AmConfigurationSetting returns the list of configuration settings for an application, service feature or component.

Syntax

Get-AmConfigurationSetting –InstanceType <string> -InstanceName <string> [-SettingName] <string>

Parameter Descriptions

[-InstanceType] <string>

Takes the type of the instance.

Required? Yes
Default value None
Accepted values ApplicationInstance

ServiceInstance

FeatureInstance

ComponentInstance
Accept Wildcards? No

[-InstanceName] <string>

Takes the name of the instance.

Required? Yes
Default value None
Accepted values Instance names, such as Terminology_Inst1 (application instance), or externalregistry (feature instance).
Accept Wildcards? No

[-SettingName] <string>

Takes the setting name to return for the instance. Only one name can be specified. If omitted, all setting are returned.

Required? No
Default value None
Accepted values Settings such as: FeaturePublisher, DiagnosticlogLevel, or Description.
Accept Wildcards? No

Example

Get-AmConfigurationSetting -InstanceType featureinstance -InstanceName externalregistry

Set-AmConfigurationSetting

Set-AmConfigurationSetting sets the configuration setting values for the application, service feature, or component.

Syntax

Set-AmConfigurationSetting –InstanceType <String> -InstanceName <String> -SettingName <String> [-SettingValue] <String> [-SettingValueFile] <String>

Parameter Descriptions

[-InstanceType] <string>

Takes the type of the instance.

Required? Yes
Default value None
Accepted values ApplicationInstance

ServiceInstance

FeatureInstance

ComponentInstance
Accept Wildcards? No

[-InstanceName] <string>

Takes the name of the instance.

Required? Yes
Default value None
Accepted values Instance names, such as Terminology_Inst1 (application instance), or externalregistry (feature instance).
Accept Wildcards? No

[-SettingName] <string>

Takes the setting name to return for the instance. Only one name can be specified. If omitted, all setting are returned.

Required? No
Default value None
Accepted values Settings such as: FeaturePublisher, DiagnosticlogLevel, or Description.
Accept Wildcards? No

[-SettingValue] <string>

Takes the new value for the setting.

-SettingFile and -SettingValue are mutually exclusive.

Required? No
Default value None
Accepted values Setting value. For example, "123".
Accept Wildcards? No

[-SettingValueFile] <string>

Takes the path to a file containing settings for the instance.

-SettingFile and -SettingValue are mutually exclusive.

Required? No
Default value None
Accepted values Setting Value. For example: c:\abc.doc.
Accept Wildcards? No
