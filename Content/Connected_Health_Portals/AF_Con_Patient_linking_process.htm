Patient signup and linking process

 

This topic describes the patient signup and linking process.

The patient signup feature integrates the patient signup and patient linking experience for patients accessing solutions via the patient site. Patients should be able to do patient linking after their initial sign up if they choose to or if the initial linking fails.

The patient performs the following tasks:

  • Signup for an account
  • Add a new record
  • Submit a patient link request

Patient signup process

  1. Patient accesses the home page and clicks signup to go to the patientsignup.aspx page.

  2. The patient signup page will use the record signup control and patient match data controls.

    The patientsignup.aspx page contains the fields that patients are asked to complete when signing up. Fields can be hidden, but not removed, to customize that page.

  3. The patient enters their account and record details and other information that could be used for the patient link process and record creation.
  4. On click of submit on the patient signup page, the page will post back to patientsignup.aspx

  5. During post back event handling, patientsignup.aspx will do the following:

  1. Store the patient link data in a cookie.
  2. Inject a JScript to trigger a POST request to the page with CREATEACCOUNT action parameter, a call back URL and along with all data submitted by patient required for record creation.
  1. takes the patient through the rest of the record creation process.
  2. Once the process is completed (or if canceled), redirects the user back to the call back URL with a status of whether the process is completed or canceled.
  3. If the process is completed, patient link request data is read from the temporary store and is submitted as a patient link request for further processing. The patient link request submission and link process happens asynchronously behind the scene while patient can view non-hospital data, if any exists.
  4. If the process is canceled, patient link data from the temporary store is disposed of and patient gets redirected to another informational page on what can be done.

Add a record

  1. Patient clicks Add new person in the record selector.
  2. Patient is redirected to addnewperson.aspx where they are asked to complete a form..
  3. Patient is asked whether the information is for a current record, for a new record, or if they don't know.
  4. Depending on the selection, the patient will be prompted to enter other required data for the patient link process.
  5. On click of submit, the page will post back to addnewperson.aspx
  6. During post back event handling, addnewperson.aspx will do the following
  1. Store the patient link data to a temporary store.
  2. Inject a JScript to trigger a POST request to a page with CREATEACCOUNT action parameter, a call back URL, and with all data submitted by patient required for record creation.
  1. takes the patient through the rest of the record creation process.
  2. Once the process is completed (or if canceled), redirects the user back to the call back URL with a status indicating whether the process was completed or canceled.
  3. The patient will be taken to a list of available records if they answered "I already have a record" or "I don't know" in step 3.
  4. If the process is completed, patient link request data is read from the temporary store and is submitted as a patient link request for further processing.
  5. If the process is canceled, patient link data from temporary store is disposed and patient gets redirected to referring page for the last selected record.

Submit a Patient Link Request

The Submit a Patient Link Request process is used when the following occurs:

  • The patient has a account and record and needs to do patient linking
  • The auto-link process was unsuccessful
  • The manual link was denied

 

  1. After successful authentication, the patient will be able to submit a new patient link request from the patient portal, if the following criteria are met:
    1. System is configured for auto-linking and patient has not exceeded the maximum allowed auto link requests
    2. System is configured for manual linking.
    3. No patient link request is pending for staff approval.
  2. Patient navigates to patientmatch.aspx.
  3. Patientmatch.aspx will use the patient match data control in non-post back mode.
  4. Enters all required data for patient link request submission.
  5. On click of submit, if the system is configured for auto-linking and patient has not exceeded the maximum allowed auto link requests, an async xml http request is used to interact with patient link service to submit the request and get the result.
  6. If the patient has exceeded the limit or if the system is configured for manual linking, after the patient link request submission, the patient will be redirected to the patient dashboard page.

One key benefit of this patient linking design is that the patient linking fields displayed in the application can handle varying amounts of patient linking data coming from the interface For example, if there are five patient linking fields on Monday, five fields show in the application. If a new field is available on Tuesday, any requests from Monday show five fields, and requests from Tuesday show six fields. The fields shown in the application are driven by the XML in the database.

If a field is not in the XML, it won’t be displayed in the application that supports patient linking.
