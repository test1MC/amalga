Application Manifest

The application manifest is an XML file that declares the identifiers, configuration settings, application dependencies, and authorization model for the hosted application. A hosted application must contain exactly one application manifest. For more information about application manifests, see Application manifests on page 1.

You cannot add assemblies to an application manifest.

The application manifest is typically defined by creating a solution containing a project based on the Amalga Application Project template. Once defined, the application manifest can be edited either manually or by using the Manifest Editor in .

Data Manifest

The data manifest is a Virtual Schema Designer (.edmx) file that declares the virtual schema used by the hosted application when accessing data. Similar to an ADO.NET entity data mode. the data manifest contains the the conceptual model, storage model, and mappings that define the read and write access that your hosted application will use through . A hosted application can optionally contain exactly one data manifest, to define the virtual schema for the hosted application and all hosted services contained by that hosted application. For more information about virtual schemas and data manifests, see Virtual schemas on page 1.

The data manifest is typically defined by adding a data manifest to a project based on the Amalga Application Project template. Once created, the data manifest can be edited either manually or by using the Virtual Schema Editor in .

Hosted Services

In , hosted services are s and s that are built on and hosted by the . A hosted application can optionally include one or more hosted services. Each hosted service consists of a service manifest and the files needed to implement the hosted service. For more information about hosted services, see Hosted services on page 1.

Service Manifest

The service manifest is an XML file that declares the identifiers, configuration settings, clustering mode, authentication mode, performance counters, and service contracts for the hosted service. Each hosted service must contain exactly one service manifest. For more information about service manifests, see Service manifests on page 1.

Service Files

A hosted service typically contains files and resources that, when compiled, provide the features and functions of that hosted service. This includes, but is not limited to:

  • Managed code to run business logic.
  • code to access data, using the virtual schema for the hosted application containing the hosted service.
  • Doors Communication Foundation (WCF) endpoints, using the Amalga Hosting Runtime API, that have access at runtime to their hosted configuration store, resolved dependencies, and other services, such as logging, auditing, and application discovery. For more information about WCF endpoints, see Endpoints: Addresses, Bindings, and Contracts, on Microsoft Developer Network at http://go.microsoft.com/fwlink?LinkID=227921.
