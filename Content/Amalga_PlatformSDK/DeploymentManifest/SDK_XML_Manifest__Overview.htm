Deployment manifest overview

 

An overview of deployment manifests, the deployment manifest resolution process, and additional information.

An integral part of the application configuration model, deployment manifests are files included with hosted applications and hosted services that declare the design-time and run-time metadata required to host an application or service in .

This topic contains the following subsections:

  • Deployment manifest types
  • Defining a deployment manifest
  • Resolving a deployment manifest

Deployment manifest types

The following types of deployment manifests are supported in hosted applications:

  • Application manifests
  • WCF service manifests
  • Doors service manifests

All hosted applications must contain an application manifest, and may contain one or more of the other types of deployment manifests, depending on the features and functionality of the hosted application.

Regardless of type, all deployment manifests provide the following general information:

  • Identifiers

    The deployment manifest contains the name, version, and unique identifier of the object declared by the deployment manifest.

    Once resolved, the deployment manifest also contains the unique name of a specific deployed instance of an object, such as a hosted application or a hosted service. Multiple instances of the same hosted application or hosted service can be supported on an Amalga topology.

  • Configuration settings

    The deployment manifest declares the configuration settings, including default values and validation rules, used by the hosted application. Configuration settings allow instances of hosted applications or hosted services to be customized as needed. For example, configuration settings could be used to define include a “Welcome text” specific to an application instance, the default size of queries returned by an application instance, and the background color of an application, configurable by an administrator.

    Once resolved,  uses this metadata to automatically create a configuration store for an administrator to manage the configuration settings for each instance of the object declared by the deployment manifest, and automatically create a page in to manage these configuration settings.

Application manifests

Application manifests provide the design-time and run-time metadata used to identify and configure a hosted application. Typically, application manifests contain the following additional metadata:

  • Application dependencies

    If the hosted application depends on the features and functionality of one or more other hosted applications, that application dependency can be explicitly declared in the application manifest. This allows to be aware of and manage these application dependencies.

    Once resolved, the application manifest contains a reference to the unique name of a specific deployed instance of a dependent hosted application. When a hosted application is deployed, provides mapping functionality to ensure that the hosted application can be linked to a specific deployed instance of a dependent hosted application, and ensures that the dependent hosted application cannot be inadvertently removed. For more information about application dependencies, see Application dependencies on page 1.

  • Test configuration

    A test configuration allows you to provide metdata, such as the location of the data host on your test Amalga topology to which you can publish the virtual schema of your hosted application, that can be used by the publishing feature in .

Resolving a deployment manifest

When deploying a hosted application, the deployment manifests for that hosted application must be updated to contain instance-specific metadata about that particular instance of the hosted application. For example, multiple instances of the same hosted application can be run on an Amalga topology, and so a unique instance name must be provided for each instance.

The metadata needed to resolve a deployment manifest depends on the type of deployment manifest.
