Service manifests

 

This topic describes the service manifest, an XML file that contains the service-specific metadata for a hosted service.

All hosted services contained in a hosted application must each contain a service manifest. A service manifest is an XML file, named AmalgaService.xml, that contains elements and attributes from the deployment manifest XML schema. For more information about the deployment manifest XML schema, see Deployment Manifest XML Reference on page 1.

The service manifest is edited by a developer building a hosted application that contains hosted services. It contains the metadata required by the to host and expose the service. The service manifest is organized around four sections, which provide the following information to the when an instance of the hosted application is debugged, deployed, or managed:

  • Service instance identification

    The service manifest contains the name, version, and unique identifier of the hosted service. Once resolved, the service manifest also contains the unique name of the specific deployed instance of the hosted service. Multiple instances of the same hosted application can be supported on a single target topology, and so each instance of the hosted services contained in that hosted application must also be uniquely identified.

  • Configuration settings

    Included in the service manifest, configuration settings provide service-specific variables that can be used to change the behavior of a hosted service. On deployment, the automatically creates a configuration store, for an administrator to manage the settings of a hosted application, and a page in the Microsoft Amalga Management Console to manage these settings for the deployed instance of that hosted application. The settings for a hosted service contained by that hosted application are also included. If default values are included, the applies those values to the settings in the configuration store during deployment. If validation rules are included, the applies those rules when changing the values for those settings.

  • Performance categories and performance counters

    After deployment, the uses the performance categories and performance counters declared in the service manifest to monitor the performance of a hosted service.

  • Clustering mode and cluster group

    For hosted services that are designed to take advantage of clustering, the service manifest provides information on the supported clustering mode and intended cluster group for a hosted service.

  • Authentication mode

    The service manifest provides information on the authentication mode required by the hosted service.

  • Logical host name

    After deployment, this attribute identifies the logical host on which an instance of a hosted service is hosted. You can specify a logical host name for debugging purposes, before deploying a hosted service.

  • Access level

    For hosted s, this attribute indicates whether the hosted service supports private or public access.

  • Service contracts

    For hosted s, the service manifest describes the service classes, service contracts, and bindings exposed by the hosted service. This information is used by the to expose and support the hosted service.
