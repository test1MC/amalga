Application manifests

 

This topic describes the application manifest, an XML file that contains the application-specific metadata for a hosted application.

All hosted applications must contain an application manifest. An application manifest is an XML file, named AmalgaApplication.xml, that contains elements and attributes from the deployment manifest XML schema. For more information about the deployment manifest XML schema, see Deployment Manifest XML Reference on page 1.

The application manifest is edited by a developer building a hosted application. It contains the metadata required by the to host the application. The application manifest is organized around four sections, which provide the following information to the when an instance of the hosted application is debugged, deployed, or managed:

  • Application instance identification

    The application manifest contains the name, version, and unique identifier of the hosted application. Once resolved, the application manifest also contains the unique name of the specific deployed instance of the hosted application. Multiple instances of the same hosted application or hosted service can be supported on a single target topology.

  • Configuration settings

    Included in the application manifest, configuration settings provide application-specific variables that can be used to change the behavior of a hosted application. On deployment, the automatically creates a configuration store, for an administrator to manage the settings of a hosted application, and a page in the Amalga Management Console to manage these settings for the deployed instance of that hosted application. If default values are included, the applies those values to the settings in the configuration store during deployment. If validation rules are included, the applies those rules when changing the values for those settings.

  • Application dependencies

    During deployment, the uses the application dependencies declared in the application manifest to confirm that other applications on which this hosted application may depend are present on the target topology. For more information about application dependencies, see Application dependencies on page 1.

  • Authorization model

    The authorization model describes all of the authorizable resources introduced by the hosted application, as well as the access levels required for a user to interact with a given resource.

No binaries can be added to the application manifest at design time.
