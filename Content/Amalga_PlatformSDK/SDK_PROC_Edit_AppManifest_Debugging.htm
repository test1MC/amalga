Edit an application manifest for debugging

 

This topic provides a procedure for editing the elements and attributes in an application manifest that are used when publishing a hosted application for debugging purposes.

Before a hosted application can be published for debugging by using the publishing feature in , several elements and attributes must be properly set in the application manifest. All of the elements and attributes that must be set prior to debugging are contained in the Test section of the Manifest Editor.

To publish your hosted application, the application manifest must have these attributes and elements:

  • A valid instance name specified for the TestInstanceName attribute of the DeploymentManifest element, to identify the test instance of the hosted application. For more information about the DeploymentManifest element, see DeploymentManifest Element on page 1. This test instance name is also used to remove any existing test instances that may have been previously published for debugging.

  • If your hosted application contains a data manifest (.edmx) file, a TestConfiguration element that contains a VirtualSchemaDataHost attribute set to the fully qualified domain name (FQDN) of the data host to which the data manifest will be published. For more information about the TestConfiguration element, see TestConfiguration Element on page 1.

    The TestConfiguration element is not included in an application manifest by default. You must use the Manifest Editor, as described in the following procedure, to include it in the application manifest if your hosted application contains a data manifest.

To edit an application manifest in for debugging

  1. In Solution Explorer, right-click the application manifest of the hosted application to be published, and then click Edit Manifest.
  2. If needed, expand Test.
  3. In TestInstanceName, type the instance name that will be used by this hosted application when published for debugging.

  4. Expand TestConfiguration, and then type the fully qualified domain name (FQDN) of the data host to which the virtual schema, if any, will be published in VirtualSchemaDataHost.

    You must perform this step if your hosted application contains a data manifest.

  5. Click OK.
