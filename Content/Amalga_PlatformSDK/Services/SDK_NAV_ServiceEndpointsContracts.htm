Service Endpoint Reference

 

This topic provides an overview of service endpoints and service contracts for the features and applications included with .

To interact with the services provided by features and hosted applications, you must use a service endpoint. You must have the following information to use a service endpoint:

  • An address that indicates where the endpoint can be found
  • A binding that specifies how a client can communicate with the service endpoint
  • A contract that identifies the operations available

Addresses

The address of an endpoint for the services provided by features and hosted applications conforms to the following template:

https://ServerFQDN:ServicePort/ServiceInstanceName/ServiceRelativeURL

Parts of an endpoint address

The address of an endpoint for a service exposed by features and hosted applications has the following parts:

  • ServerFQDN - the fully qualified domain name of the server that hosts the WCL service.

  • ServicePort - the TCP port assigned to the site hosting the WCL service. The following table lists the default service ports for the sites used by to host features and applications.

    Site Service port
    Microsoft Amalga Central Administration 3.0
    		 9999
    Microsoft Health Hosted Services 3.0 8801
    Microsoft Health Physical Host Services 3.0 8900
    Microsoft Health Platform Services 3.0 8889

    By default, features are hosted on the Microsoft Health Platform Services 3.0 site, while hosted applications and hosted services, including platform applications and services, are hosted on the Microsoft Health Hosted Services 3.0 site.

  • ServiceInstanceName - The instance name assigned to the web service when it was deployed. Instance names are site-specific and are usually specified in the service manifest. In practice, many instance names are constructed by adding a standard postfix (such as "_Ins1") to each service name.

  • ServiceRelativeURL -The path to the .svc WCF implementation file for each service. The structure of the relative URL varies depending on the service.

Example

For example, if the task flow service were deployed to Webserver.Example.com and its instance name was derived from its service name as described above, its default service endpoint would be:

https://Webserver.Example.com:8801/TaskFlowService_Ins1/Microsoft.Amalga.TaskFlow.Service.TaskFlowService.svc

Bindings

Bindings describe how to communicate with an endpoint for a service. For example, bindings can indicate the transport protocol used by the endpoint, or the encoding to be used for messages sent to and from the service endpoint.

Several binding options are supported for service endpoints exposed by features and hosted applications. For more information about available binding options for services exposed by hosted applications, see Bindings Element on page 1.

Contracts

Contracts describe the functionality exposed to a client by an endpoint for a service. This includes the operations that can be performed, the form of the message used to perform an operation, as well as the input parameters or output responses expected by the endpoint.

Endpoints and contracts for features

The endpoints and contracts for the services exposed by features, such as the Logger and Topology features, allow access to integral components of the platform.

For more information about the endpoints and contracts for features, see Endpoints and contracts for features on page 1.

Endpoints and contracts for platform applications

The endpoints and contracts for the hosted applications included with , such as the DataConnector and Terminology applications, allow access to applications that expand the capabilities of the platform.

For more information about the endpoints and contracts for platform applications, see Endpoints and contracts for platform applications on page 1.
