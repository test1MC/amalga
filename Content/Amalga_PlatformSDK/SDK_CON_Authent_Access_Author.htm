Access, authentication, and authorization

 

This topic briefly discusses three features provided when using the Amalga hosted application framework, and displays an code example from a sample application manifest.

Access, authentication, and authorization are controlled through the application manifest of hosted applications. The manifest defines access levels, where roles are mapped to those access levels. Then the manifest specifies which resources (services and application views) are assigned to which access levels.

provides ways for you to control access, authentication, and authorization for data that your application will use. For hosted applications, these are defined in the application manifest. You start by defining access roles, as described in Manage roles. Then, in the application manifest, you specify access levels, which contain one or more roles. Finally, for each resource, which may be either a service or an application view, you specify which access levels are authorized to access that resource.

The following example shows an <ApplicationAuthorization> element from a hosted application's application manifest.

<ApplicationAuthorization>
  <AccessLevels>
    <AccessLevel name="AllAccess">
      <Role name="BUILTIN\AmalgaSysAdmin"/>
    </AccessLevel>
    <AccessLevel name="OfflineAccess">
      <Role name="BUILTIN\AmalgaService"/>
    </AccessLevel>
  </AccessLevels>
  <Resources>
    <Resource type="Service" name="AuthorizationSvc" operation="GetApplicationRoles">
    <AccessLevel name="AllAccess"/>
    <AccessLevel name="OfflineAccess"/>
   </Resource>
  <Resource type="Service" name="AuthorizationSvc" operation="GetRoles">
    <ApplicationRole name="AllAccess"/>
    <ApplicationRole name="OfflineAccess"/>
  </Resource>
  <Resource type="AppView" name="AppPrivateResource" operation="ReadWrite">
    <ApplicationRole name="AllAccess"/>
  </Resource>
  </Resources>
</ApplicationAuthorization>
