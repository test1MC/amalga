platform development architecture

 

This topic describes Amalga data flow and a high level platform architecture. Data comes from multiple feeds, and the Amalga extraction process creates a database called the Amalga data asset. Data is processed for performance, tagging, and other requirements. Data is the accessed by applications and services through the virtual schema. Applications use a service layer that can consist of Amalga Platform and Application services, as well as custom services. They can also be built on the to make use of additional services and controls.

In the Amalga platform, data begins as a collection of feeds, and is aggregated into a central data store. Applications and services then access data through their own virtual schema. Amalga provides a variety of features to assist in building applications that use these virtual schemas, including services and controls.

Data flow

Once the data is in the Amalga data asset, applications and services can access it through virtual schemas, as shown in the following diagram. Data within the data asset is processed for performance, removal of identification (for copies of the data used in research), and tagging with standard codes used by the terminology service. Each application and service contains its own virtual schema, which specifies which of the Amalga entities in the data asset the application will use.

Platform Architecture

Once a virtual schema is designed for an application or service, then you access your data entirely through this abstraction layer. The following diagram displays the various pieces of the Amalga platform, and how they fit together.

 

Between the applications and the virtual schema is a service layer. This layer consists of three parts:

  • Amalga platform and application services
  • Amalga platform applications
  • Custom services that you develop
  • Services that are a part of the

Platform services consist of all services required for the platform to run which must be installed on the topology server, and application services are services required for the platform to run which are installed on any server in the topology. Platform services include the topology service, the logger service, the auditing service, the security service, the virtual schema service. Application services are primarily the physical host service, which allows hosting applications on a physical server and adds this physical server to the logical host.

Amalga platform applications are hosted applications that provide functionality that you can use in your custom applications. Examples include the data connector, the terminology service, and the eventing service.

The custom services that you develop may be WCF services or Doors services. These can be created using the Visual Studio Amalga project templates and the virtual schema designer.

The custom applications that you develop may be Amalga hosted applications, or could be any kind of application that uses the services. There are many advantages to hosting your applications, but it is not a requirement. The architecture is designed flexibly so that you can use many types of technology to build applications, such as Silverlight and ASP.NET.
