controls overview

 

This topic describes the controls provided by the and the common features that those controls provide.

The controls provided as part of the are the building blocks for Web-based applications. Web sites that include controls have access to a rich set of common features that enhance the ability of the controls to interact with each other and with the platform services.

controls are not just passive consumers of the platform services. Many of the common features of the Framework controls depend on client side code that is loaded into the local browser by the controls themselves. Managing those features presents important design considerations when placing them into applications, modifying Amalga production controls, and creating custom controls.

Web sites built with controls also begin with a standard look and feel, illustrated by the basic sites installed as part of a default implementation.

Categories of control

The controls in the SDK can be divided into two groups.

  • Production controls - most of the controls that comprise the basic portal sites are included in the SDK for reuse or extension. Most of these controls are also available in the .
  • Generic Controls - The SDK includes base classes for creating new controls. Controls that inherit from these base classes have access to the features common to all controls.

Features of a control

The following features are available to Web sites built with controls:

  • Page messaging - the page messaging system enables the different elements of a Web site to exchange messages asynchronously, without requiring a postback to the Web server. This allows multiple page elements to coordinate their content in response to a user interaction with a single element. The page messaging system also includes support for HL7 CCOW context passing and management, within the scope defined by a single browser window.
  • Notifications - the notification system delivers event-driven messages to the user, where the content of the message is determined by the meaning of the event and not by the current context of the user session. Notifications are automatically surfaced to the user using the SharePoint UI.
  • data access - data-bound controls have access to a specialized data source that exposes some or all of the content of an associated .
  • Cross application task flow - the task flow system provides the infrastructure for a control in one application to request action from another application, or from itself in a different user context. Tasks can be defined with multiple states, allowing one or more applications to track and advance each task from initiation to completion.
  • Logging - the logging system allows applications to persist information to the logs in response to meaningful events, for both server-side and client-side events.
  • User settings and configuration - the Configuration system allows controls to store and retrieve strongly typed values on a per-application basis. The system also allows those values to be overridden on a per-user basis.
  • Application identity - controls can resolve the identity of their own application instance.
