Access inheritance

 

This topic introduces the concept of access inheritance for entities.

Configuring access inheritance allows you to reuse access rules configured for an entity. With access inheritance, you can apply an entity's access policy to a child or peer entity, or to an application-private table.

For example, if the Contoso.Visit entity is configured to inherit access from the Contoso.Patient entity's access policy “as parent,” then Contoso.Visit is a child entity of Contoso.Patient (and Contoso.Patient is a parent entity of Contoso.Visit). If the Contoso.Visit entity is also configured to inherit access from the Eastlake.Visit entity's access policy, but this time “as peer,” then Contoso.Visit and Eastlake.Visit are peer entities.

Every entity has an access policy, which includes role-specific access rules and role-specific permissions for any sensitive labels applied to properties of the entity. A rule on a parent entity can be specific to children entities only, in which case the rule does not grant access to the parent entity. In addition, rules can be defined for children entities that restrict access beyond any inherited access policies.

A parent rule, or rule that is set for a parent entity, dictates the maximum privilege of the role; the child rule further restricts access. For a given role, rules are applied across peers and from parent to child entities with an AND condition.

Parent-child relationships

Rules configured for a given role on a parent entity apply to the child entity as well. You can configure additional rules for that role on the child entity to further restrict access, using an AND condition.

For example, if the role Doctors has access to the entity Patient where Patient.VIPFlag != ‘1’, and the entity Visit is configured as a child of Patient, then members of the role Doctors can access visit records of patients whose VIPFlag value is not ‘1’.

In addition, you can configure a rule for the role Doctors on the child entity Visit to further restrict access to visit data. For example, you can set a rule for Doctors to limit access to visits where Visit.DischargeDateTime IS NOT NULL. In this case, members of the role Doctors can access visit data that meet the following condition:

Where Patient.VIPFlag != ‘1’

AND Visit.DischargeDateTime IS NOT NULL

Configuring a parent-child entity relationship involves explicitly defining a primary key-foreign key relationship between the entities. For more information, see Configure access inheritance for an entity.

Peer relationships

Peer entities share their rules.

For example, if the role Doctors accesses the entity Contoso.Patient under the read-only rule where VIPFlag != ‘1’, and the entity Eastlake.Patient is configured as a peer to Contoso.Patient, then members of the role Doctors can access patients from Eastlake.Patient whose identifiers correspond to Contoso.Patient.VIPFlag != ‘1’. Note that in this example, Eastlake.Patient does not itself have a VIPFlag rule for the role Doctors.

In addition, you can configure rules for the role Doctors on the entity Eastlake.Patient. In this case, the rules for Eastlake.Patient apply to Contoso.Patient. For example, you can configure an additional rule for the role Doctors on Eastlake.Patient where Eastlake.Patient.Consent = ‘1’. In this case, members of the role Doctors accessing data from Eastlake.Patient or Contoso.Patient can read data from any patient whose identifier meets the following condition:

Where Contoso.Patient.VIPFlag != ‘1’

AND Eastlake.Patient.Consent = ‘1’

Configuring a peer relationship for two entities involves explicitly identifying and matching the primary keys for both entities. For more information, see Configure access inheritance for an entity.
