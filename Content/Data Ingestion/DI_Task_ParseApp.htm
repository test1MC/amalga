Resolve manifests

 

This topic describes how to resolve application and service manifests.

Pipeline data parsing is an activity that selectively gets shredded columns with which it can derive a new column, collapse a view, transform data, apply a lookup function, or map enterprise identifiers. For a functional profile, see Data Parser application.

Application manifest

To resolve an application manifest

  1. Copy, rename, and edit an example file.
  2. Replace the deployment-manifest instance name with a new name that uniquely identifies the application instance. For help, see DeploymentManifest element.

  3. Replace the dependency instance name with the instance name of the local Terminology application. 

    <Dependencies>
        <Dependency Name="TerminologyAppDependency" InstanceName="TerminologyApplication_Ins1" 
                    Type="Application" Id="c031033a-49be-4c8f-a6a8-cb95bddaac0a" MinVersion="1.0.0.0" />
</Dependencies>

    Get the local terminology instance name from the Amalga Management Console.

  4. Save the file.

Service manifest

To resolve a service manifest

  1. Copy, rename, and edit an example file.
  2. Replace the deployment-manifest instance name with a new name that uniquely identifies the service instance. For help, see DeploymentManifest element.

  3. Replace the logical host name with the local name. For help, see DoorsWorkerManifest element.

    Get the local logical-host name from the Amalga Management Console.

  4. Replace the Type element value with the value used in the related feed-acquisition service manifest to define the MsgType element. For example, if the feed's message-type value is ABC, then the parser's feed-information type must be ABC. For more feed-related options, see FeedInfoXML setting.
  5. Assign message shredder input values for all segment elements that you want to parse. For help, see MessageShredderInput element, Segment element, and Elements element.

  6. Assign data-parser output values for each new derived column, collapsed view, transformed data, or lookup function. For help, see DataParserOutput element, Segment element, Elements element, CollapseConfiguration element, Plugins element, Transform element, and Lookup element.

    Segment elements within <MessageShredderInput> and <DataParserOutput> element tags do not have the same behavior. Specifically, you must never include transform actions in segments in the <MessageShredderInput> section. You can only specify transform actions in segments in the <DataParserOutput> section.

  7. (Optional) Define one or more data constants to use in the declaration of other segment elements. For help, see DataConstants element.
  8. (Optional) Configure enterprise identifiers. For help, see Map enterprise identifiers and Configure merge actions.
  9. (Optional) Update the attributes of an existing error policy. For help, see ErrorHandlingConfiguration element.
  10. (Optional) Declare a custom plug-in application. For help, see Add parser extensions.
  11. Save the file.
