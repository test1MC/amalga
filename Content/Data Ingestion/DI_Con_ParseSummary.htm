Remarks

Once shredded data is understood and interpreted, the can be configured to create derived columns for cleansing data, normalization, code lookups, composite transforms like concatenations, and so on. The data parser is responsible for generating these columns. Each derived column should include a terminology concept identifier when the message type supports semantic tags.

Context key

A context key is a combination of fields that are native to a message and uniquely identify an entity. As a general rule, for every entity that you want to create out of a message type, you must specify one context key.

In addition, a context key eliminates partial information from an output view (called a collapsed view). For example, if you have multiple records for a patient, the parser can collapse the set into a single record showing only the relevant information determined by the context key.

Data transforms

Configuration-based transformation makes it easy for you to prepare shredded data for use by applications. Supplied transforms range from simple string manipulation, such as adding title capitalization, to generating new fields by concatenating existing columns into new columns. The pipeline contains a transform library that you can include as a plug-in to your parser. The library contains MessageHelpers. For configuration details, see Resolve Data Parser manifests.

In addition to the provided libraries, you can import your own custom transform logic as an application plug-in.

options

maps external, source-specific identifiers to a set of internal identifiers across multiple visits, institutions, and domains. These identifiers are unique within and are collectively processed by the . You can map non-standard source identifiers to standardized elements directly in a parser service manifest; no external code is required.

Each correlation element maps to two pieces of information that the engine concatenates into a single identifier. First, the Id element must contain values for the SourceTable and SourceName attributes. These values are typically provided as message shredder input. Second is the Domain element, typically the assigning authority for the first piece of information. You can define each domain name as a separate constant in your manifest. It is possible to use shredded input to define the domain.

The following table shows the four correlation elements and provides identifier-value examples. The elements are hierarchical: PersonIdentifier is the highest level. Map only the elements that you need. For example, if MRN and Account are sufficient for a feed, then map only the PatientRecordIdentifier and EncounterSetIdentifier elements.
